Visuals Made Sane

Tagged as en, toolbox, visual
Written on 2020-10-01

Almost every time I had to produce a visual representation for - literally - anything, it used to be disturbing. A balance between control, speed, accuracy and easiness seemed unreachable even for a routine plotting of data at hand. Well, not anymore!

A brief research lead me to a conclusion that the sanest way to draw something fast, efficient and nicely-looking these days is to use that very piece of software you are probably already running.

Yes, I mean the mighty web browser. Some observations follow.


Presentations: Reveal.js

Though I like the idea of the beamer LaTeX package of programmable presentation files, I hate the package itself for the tremendous amount of effort it requires to output a PDF that won't cause your audiences' eyes bleed. Yes, out-of-the-box beamer is plain ugly. Just fixing its looks would take days for an unprofessional layout designer.

Forget it. Use PowerPoint or LibreOffice Impress, or any other WYSIWYG editor. Everything goes for this ungrateful task, especially if you're short on time. But in case you don't have to rush and know a bit of markup, have a glance at reveal.js.

Under the hood it's the same idea of using hypertext instead of a visual editor for presentation outline. Yeah, like beamer, only that:

  • it looks way better from the start;
  • markup language is same as for the web;
  • it has all the key features of visual editors.

The last is true when reveal.js is served through npm, bringing even familiar speaker's screen with previews, notes and timer. Though this goes as unnecessary "extra". To build a presentation with reveal.js one really needs only a browser and text editor, meaning zero installation overhead.

With support for LaTeX formula notation built-in through MatJax, and simple but elegant block layouts available with use of modern css, I'd say that reveal.js is "more beamer than beamer".

Plots & Charts: Vega-Lite

Plotting was a cursed subject to me where no "golden point" tools seemed to be found. I wont' bother naming cons of gnuplot, matplotlib, plplot etc, each of them has a bunch. And the main problem is that with any of the popular plotters charting just simply takes a lot of time. Nothing handy seemed to appear outside of rather enclosed environments like Mathematica or Racket's plot library (one of the cool features of Racket, a language that is an ivory tower of its own). With standalone plotters it takes too much red tape to make a chart fine enough to be shared. Even with all the data at hand. It hurts.

All I want is an accurate, powerful, portable solution with concise and understandable semantic to draw them plots from raw data and from what is piped from my programs as well. Is it too much to ask in 2020? Well, in fact a good graphics grammar design is a non-trivial topic, with books and papers published.

The paper from the latter link describes the design of one such grammar system - Vega-Lite. I discovered it accidentally, but fell in love at once.

According to the paper:

Vega-Lite combines a traditional grammar of graphics, providing visual encoding rules and a composition algebra for layered and multi-view displays, with a novel grammar of interaction.

It allows to specify compositions of charts on a canvas, interactive selections and data transforms. Vega-Lite (coupled with low-level underlying Vega engine) can display quite sophisticated styles of charts (3-dimensional plots are on the go), requiring rather short specification from a user. I'd say, go read the design paper for details, then address the reference guide.

Note first that the whole plot spec is just a JSON. Here it includes the data points too, but in principle the data can be served from a separate csv-file. That would leave here only the plot view spec (from line 59) that is short, readable and in general makes sense.

Mind that round button in the right top corner for the chart export options. One of it says "Open in Vega Editor". It spawns an interactive editor in the browser for that very plot you are looking at.where the whole plot view can be edited live. Great to tackle with different plotting styles!

So, creating a plot as easy as editing a simple html-template with Vega-Lite spec. One can go further, serving a trivial single-page web app with the plot spec and data assembled and serialized as JSON (a task for which plenty of libs are available everywhere).

That app can be improved again a little bit, inspired by vega-lite with websockets example. That brings fully dynamic auto-updating charts! That's how I use Vega coupled with my computational workflows, having the plot updated after each step processed with our Quantum-Mechanical heavy artillery. One may think about live dashboards powered with only a Raspberry Pi running a browser. I can't stop thinking about them now. I do all the time...


In this test run the Heat Flux components from the two DFT programs get straight onto the plot right as each Molecular Dynamics step is processed.

3D scenes: WebGL

For everyone trying to enter low-level 3D rendering nullprogram's advice is to start directly with WebGL, since it utilizes the most modern standards and has very low overhead. Again, only a modern web browser is needed - no need to drag a GUI window library. Portability is also high. Android devices do render WebGL, hell even single-board computers do render!

The author of the best WebGL tutorial I've seen so far also wrote an utility library TWGL that helps to set up the scene components and start drawing faster (without hiding any low-level parts of the WebGL API).

This brings you GL, and fast, but what about interactiveness? Remember that this framework runs entirely in browser. The program that can use javascript to control the DOM of pages it shows. So with access to the javascript console one could rebuild in parts or in whole an appearance of a page, e.g. what is rendered on its <canvas> element.

This idea is shown in this demo (I mirror the video below), where the author sends instructions from the Common Lisp REPL directly into his browser as Js instructions. Lisp forms are translated into javascript code using Parenscript library and sent as messages through websocket to a client page. Though it's a universal trick to inject any code on a page from any other programming environment (not necessarily Lisp), applied to WebGL canvas it creates an effect of manipulating a 3D scene without reloading anything!

And that last part can give a spin to a various tasty hobby projects... But those ideas I'll keep for myself for now. Okay, just a hint one more last time.
